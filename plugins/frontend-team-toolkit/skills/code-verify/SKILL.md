# Code Verify — 先锚定，再迭代

> **版本**: 2.0.1  
> **定位**: AI 辅助第三方开发的方法论 + 可执行技能（非单纯 Linter，非代码生成器）  
> **边界**: 验证与决策支持；最终判断权在人；运行时联调仍须人工或小步实测

---

## 一、哲学内核

### 1.1 铁律

**在让 AI 写代码之前，先确认方向是对的。**

方向错了，迭代越多离正确答案越远——这叫**迭代幻觉**：导航导错路，每开一公里都在偏离，却一直在「重新规划路线」。

### 1.2 三个陷阱（为何要本技能）

| 陷阱 | 表现 | 本技能如何应对 |
|------|------|----------------|
| **信息不对称** | 你和 AI 都不真正懂该 API，互相确认彼此的不确定 | 强制对照**官方文档 + 参考实现**双锚，不靠概率共识 |
| **迭代幻觉** | 「改一轮再测」默认方向正确 | **置信度闸门** + **止损点**，低分时禁止盲迭代 |
| **自信幻觉** | AI 语气笃定，你倾向相信 | 结构化报告拆成 ✅/⚠️/❌，强制暴露不确定项 |

### 1.3 三个「不」

1. **不完全信任** — AI 输出必须经过锚定验证  
2. **不盲目迭代** — 方向错了，改再多也没用  
3. **不放弃判断** — 你是配对编程里「看方向」的人，AI 只负责「敲键盘」

### 1.4 适用范围（工具无关）

本技能适用于**任何**具备以下条件的第三方工具/SDK/API：

- 有官方文档或可核验的参考实现  
- AI 可能生成「看起来对、跑起来错」的接入代码  

`examples/` 目录下的具体案例（如某 IM SDK、支付 API）仅为**哲学应用示例**，不是技能默认场景。  
**Agent 不得**在未获用户上下文时，默认假定某一特定 SDK 或套用某一案例的 API 名称。

## 二、全链路闸门（五阶段）

本技能覆盖从「收到任务」到「经验沉淀」的完整链路，**超越**「只验一段代码」：

```
收到任务
  ↓
【阶段 1】方案锚定 — 先要方案，不要代码
  ↓
【阶段 2】双锚验证 — 官方文档 + 参考实现（七维引擎）
  ↓
【阶段 3】小步验证 — 最小可验证单元，写完即测
  ↓
【阶段 4】止损换策 — 时间盒 + 置信度闸门
  ↓
【阶段 5】经验沉淀 — 检查清单归档
  ↓
交付 / 进入下一模块
```

**默认触发时机**（按优先级）：

1. AI 给出**技术方案**后、写代码前 — **必跑阶段 1–2**  
2. AI 给出**首段代码**后 — **跑阶段 2–3**  
3. 联调反复失败 — **触发阶段 4**，禁止无脑「再改一轮」  
4. 模块结束 — **阶段 5** 落盘

## 三、阶段 1：方案锚定

### 3.1 第一个 Prompt（不要「帮我写代码」）

```
请帮我梳理 [具体需求] 的技术方案。

要求：
1. 列出需要调用的核心 API 或 SDK
2. 给出每个 API 的官方文档链接（须可访问）
3. 用文字描述调用流程（含初始化顺序）
4. 列出可能的坑和注意事项
5. **必须标注你不确定的部分**（不得用确定语气掩盖）
6. 建议的最小可验证单元拆分（按官方文档章节顺序，不要按你臆想的代码结构）
```

### 3.2 阶段 1 通过标准

- [ ] 方案含官方文档链接，且非第三方博客  
- [ ] 存在明确的「不确定」标注（若无，视为红旗，要求 AI 补充）  
- [ ] 调用顺序与文档「快速开始 / 接入步骤」章节可对应  
- [ ] 尚未进入大规模写代码

### 3.3 阶段 1 失败处理

方案含糊或缺少文档链接 → **禁止进入阶段 3 写代码**，回到 3.1 重问或人工补文档。

## 四、阶段 2：双锚验证（七维引擎）

### 4.1 双锚定义

| 锚 | 作用 | 来源优先级 |
|----|------|-----------|
| **文档锚** | 规范真相：API 名、参数、认证、顺序、版本 | 官方文档 > 官方 SDK README |
| **实现锚** | 可运行参照：真实项目怎么调 | 官方 example > 官方技术博客示例 > 高 star 开源（交叉验证） |

**没有参考实现的开发，像没有地图的导航**——文档锚告诉你「规范说什么」，实现锚告诉你「别人怎么活下来的」。

### 4.2 七维验证（文档锚）

| # | 维度 | 检查内容 |
|---|------|----------|
| 1 | 📄 文档存在性 | 链接可访问、官方域名、标题匹配 |
| 2 | 🔑 API 签名 | 方法名、参数名/类型/必填、返回值、端点 |
| 3 | 🔐 认证方式 | Token/OAuth/Key、参数名、流程完整性 |
| 4 | 📦 依赖包 | 包名、版本、安装命令是否真实存在 |
| 5 | 🏗️ 初始化顺序 | 配置→注册→注入→调用的顺序是否与文档一致 |
| 6 | 🔄 版本兼容 | 是否使用废弃 API；SDK 与文档版本是否匹配 |
| 7 | ⚡ 语法检查 | 静态语法（**不能**替代运行时联调） |

### 4.3 参考实现锚（阶段 2 扩展，必做）

在七维之外，**至少找 1 份参考实现**并完成对比：

```
参考实现对比清单：
- [ ] 来源是否官方或可信（见 4.1 优先级）
- [ ] 初始化入口是否与 AI 方案一致（如：SDK 初始化 → 鉴权/配置 → 首次 API 调用的先后顺序）
- [ ] 核心 API 调用签名是否与文档锚结论一致
- [ ] 错误处理 / 回调模式是否与官方 example 同构
- [ ] 若与 AI 方案不一致：以文档锚为准，实现锚作辅证；二者冲突则 ⚠️ 存疑并人工裁定
```

**搜索参考实现**：

- 官方 SDK 的 `example` / `demo` 目录  
- 官方文档内嵌代码块与独立「示例」页  
- GitHub 官方组织仓库  
- Stack Overflow 高赞（**必须**与文档锚交叉验证，不可单独采信）

### 4.4 阶段 2 执行步骤

```
① 收集输入（工具名、方案/代码、语言、已知 doc-url、可选 ref-url）
② 搜索/确认官方文档 → web_search / web_fetch
③ 提取文档摘要（版本、API 页、认证页、安装页）
④ 定位参考实现 → 官方 example 优先
⑤ 逐项七维验证 + 参考实现对比
⑥ 生成报告（templates/report-template.md）
```

**收集输入模板**：

```
📋 锚定验证任务

1. 工具名称: {工具名}
2. AI 方案/代码: {内容或文件路径}
3. 编程语言: {可选}
4. 官方文档 URL: {可选，有则加速}
5. 参考实现 URL: {可选，无则 Agent 搜索}
6. 当前阶段: 方案 / 首段代码 / 联调受阻
7. 已消耗时间: {可选，用于止损判断}
```

### 4.5 各维度执行要点

**维度 1–3、5–6**：从文档提取后与 AI 方案逐字段对比；无法访问标 ⚠️ 而非 ❌。  
**维度 4**：`npm view` / `pip index versions` 等查询包管理器。  
**维度 5**：**按官方文档章节顺序**拆步，不按 AI 生成的文件结构拆。  
**维度 7**：见文末「语法检查命令速查」；通过仍须阶段 3 运行时验证。

## 五、阶段 3：小步验证

### 5.1 原则

**每次只写一个最小可验证单元（MVU），写完立刻测，通过才进入下一步。**

一次性写完全部再测 → 出错时无法区分是 API、参数还是整体思路错误。

### 5.2 MVU 拆分规则

1. 以**官方文档章节顺序**为拆分依据，不以 AI 代码模块为准  
2. 每一步能在 **5–15 分钟**内给出明确通过/失败信号  
3. 每一步只引入**一个**新的未知假设（如新 API、新权限、新环境）

### 5.3 失败协议（禁止 AI 直接甩新代码）

某步失败时：

1. 把**完整**错误信息（堆栈、网络响应、控制台）贴给 AI  
2. 要求 AI **先分析原因**，再给方案  
3. 你确认分析合理后，才测新方案  
4. 若同一 MVU 连续失败 **≥2 次** → 触发阶段 4 止损

### 5.4 阶段 3 与七维的关系

| 七维 | 阶段 3 作用 |
|------|------------|
| 1–6 | 已在阶段 2 完成；阶段 3 不再重复，除非代码偏离方案 |
| 7 | 写代码后快速语法检查 |
| — | **运行时结果**为阶段 3 独有：通过/失败/部分通过 |

阶段 3 输出：**MVU 进度表**（见报告模板 §小步验证进度）。

## 六、阶段 4：止损与换策略

### 6.1 时间盒（个人经验值，非官方标准，可按团队微调）

| 熟悉度 | 建议上限 |
|--------|----------|
| 熟悉技术栈 | 30 分钟 |
| 半生不熟 | 45 分钟 |
| 完全陌生 | 60 分钟 |

超时且 MVU 未通过 → **禁止**继续「让 AI 再改一轮」。

### 6.2 置信度闸门（与七维联动）

| 通过数 | 置信度 | 决策 |
|--------|--------|------|
| 7/7 | 100% | ✅ 可进入阶段 3 小步写码 |
| 6/7 | 86% | ✅ 可进入，须处理存疑项 |
| 5/7 | 71% | ⚠️ 人工确认存疑项后再写码 |
| 4/7 | 57% | ⚠️ 修正方案后再写码；**禁止**大规模实现 |
| ≤3/7 | ≤43% | ❌ **触发止损**；方向可能错误，须换策略而非迭代 |

### 6.3 换策略（三选一或组合）

**策略 A — 官方社区**：论坛、GitHub Issues、Stack Overflow 搜同类问题  
**策略 B — 真人求助**：技术群、同事、曾做过该对接的人  
**策略 C — 换方案**：当前 API 路径不通，评估替代路径（如降级接口、服务端代理、其他官方推荐集成方式）

止损不是放弃，是**防止沉没成本扩大**。

---

## 七、阶段 5：经验沉淀

每次任务结束（无论成败），更新检查清单：`templates/checklist-template.md`。

**最低归档**：

- 工具名 + 文档 URL + 参考实现 URL  
- 本次七维置信度 + 实际联调结果是否一致  
- 踩坑点（尤其初始化顺序、认证、环境差异）  
- 下次同类任务的 MVU 拆分建议  

清单价值：把个人经验固化为团队资产，避免重复踩坑。

## 九、技术实现

### 9.1 Agent 工具

```
web_search  → 搜官方文档、参考实现
web_fetch   → 抓取文档与 example
browser     → web_fetch 失败时
exec        → 语法检查、依赖查询、可选最小复现脚本
```

### 9.2 语法检查命令速查

| 语言 | 命令 |
|------|------|
| JavaScript | `node --check file.js` |
| TypeScript | `npx tsc --noEmit file.ts` |
| Python | `python3 -m py_compile file.py` |
| Go | `go vet ./...` |
| Java | `javac -d /tmp file.java` |

### 9.3 依赖包查询

| 包管理器 | 命令 |
|---------|------|
| npm | `npm view {package} version` |
| pip | `pip index versions {package}` |
| Go | `go list -m -versions {package}` |

### 9.4 脚本入口

```bash
bash plugins/frontend-team-toolkit/skills/code-verify/validate.sh \
  --tool "工具名" \
  --code-file path/to/code.js \
  --lang javascript \
  --doc-url "https://..." \
  --output report.md
```

脚本生成**全链路验证骨架**；Agent 须按本 SKILL 填完五阶段与七维结果。

## 十二、Agent 执行自检

任务结束前核对：

- [ ] 阶段 1：方案含文档链接与不确定标注  
- [ ] 阶段 2：七维报告 + 至少 1 份参考实现对比  
- [ ] 阶段 3：MVU 进度已记录（若已进入写码）  
- [ ] 阶段 4：置信度 ≤4/7 或超时 → 已建议止损/换策略，未盲目迭代  
- [ ] 阶段 5：已提示更新检查清单或已写入报告附录  
- [ ] 报告结论含：置信度、是否建议继续、下一步动作  
- [ ] **未**在用户未指定时默认套用 `examples/` 中某一 SDK 的 API 名称或调用顺序
