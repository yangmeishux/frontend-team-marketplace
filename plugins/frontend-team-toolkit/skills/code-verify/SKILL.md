# Code Verify Skill - 第三方工具代码验证器

> **版本**: 1.0.0  
> **创建日期**: 2026-05-19  
> **定位**: 自动验证 AI 给出的第三方工具代码/方案是否正确  
> **覆盖**: 7 个通用验证维度，适用于任何有公开文档的工具

---

## 🎯 技能定位

**解决什么问题？**

当你让 AI 写你不熟悉的第三方工具代码时，AI 可能从第一步就错了。你反复测试、反复改，浪费大量时间。

**Code Verify 做什么？**

在 AI 写代码**之前**或**之后**，自动验证方案的正确性，给出结构化验证报告。

**不做什么？**

- ❌ 不是代码生成器
- ❌ 不是 Linter
- ❌ 不替代人工最终判断

---

## 📋 7 个通用验证维度

不管什么工具，AI 给出的方案都可以被这 7 个维度验证：

| # | 维度 | 检查内容 | 验证方法 |
|---|------|----------|---------|
| 1 | 📄 文档存在性 | API/文档链接是否真实存在 | 抓取 URL，检查 404 |
| 2 | 🔑 API 签名对比 | 方法名、参数、返回值 | 提取文档签名，逐字段对比 |
| 3 | 🔐 认证方式 | 认证流程/参数 | 提取文档认证章节，对比实现 |
| 4 | 📦 依赖包检查 | 包名、版本号、安装方式 | 查询包管理器（npm/pip/...） |
| 5 | 🏗️ 初始化顺序 | 工具使用顺序 | 提取官方示例，对比调用链 |
| 6 | 🔄 版本兼容性 | SDK/API 版本是否匹配 | 检查文档版本标签、release notes |
| 7 | ⚡ 语法检查 | 代码能否通过语法检查 | 运行对应语言的 linter |

## 🚀 使用流程

### 标准流程

```
① 收集输入 → ② 搜索文档 → ③ 提取信息 → ④ 逐项验证 → ⑤ 生成报告
```

### 步骤详解

#### 步骤 1: 收集输入

向用户确认以下信息（能从上下文推断的不问）：

```
📋 代码验证任务

1. 工具名称: {工具名}（如：企业微信 JS-SDK、Stripe API）
2. 代码/方案: {AI 给出的代码或方案描述}
3. 编程语言: {语言}（可选，用于语法检查）
4. 官方文档: {URL}（可选，已知时提供可加速）
```

#### 步骤 2: 搜索官方文档

**目标**: 找到工具的官方文档页面

**方法**:
1. 如果用户提供了文档 URL → 直接使用
2. 否则，使用 `web_search` 搜索：`{工具名} official documentation API reference`
3. 优先选择官方域名（非第三方博客/教程）
4. 记录找到的文档 URL

**搜索关键词策略**:

| 工具类型 | 搜索关键词 |
|---------|-----------|
| JavaScript/Node.js | `{tool} npm documentation API` |
| Python | `{tool} pip documentation API reference` |
| REST API | `{tool} API documentation endpoints` |
| SDK | `{tool} SDK documentation getting started` |
| 企业级工具 | `{tool} developer documentation` |

#### 步骤 3: 提取文档关键信息

**目标**: 从官方文档中提取以下信息

使用 `web_fetch` 抓取文档页面，提取：

```markdown
## 文档信息摘要

- 文档 URL: {URL}
- 文档版本: {version}
- 最后更新: {date}
- API 参考页: {URL}
- 快速开始页: {URL}
- 认证说明页: {URL}
- 安装/依赖说明: {URL}
```

#### 步骤 4: 逐项验证（7 个维度）

对每个维度执行验证，记录结果。

**维度 1: 文档存在性**

```
检查项:
- AI 引用的文档链接是否可访问
- 页面标题是否匹配工具名称
- 是否为官方文档（非第三方）

结果: ✅ / ⚠️ / ❌
```

**维度 2: API 签名对比**

```
从文档提取:
- API 方法名称
- 参数列表（名称、类型、是否必填）
- 返回值类型
- 请求方法（GET/POST/...）
- 端点路径

对比 AI 代码:
- 方法名是否一致
- 参数数量和名称是否匹配
- 参数类型是否匹配
- 返回值处理是否正确

结果: ✅ 完全匹配 / ⚠️ 部分匹配 / ❌ 不匹配
```

**维度 3: 认证方式**

```
从文档提取:
- 认证类型（Token/OAuth/API Key/...）
- 认证参数名称
- 认证流程步骤
- Token 获取方式

对比 AI 代码:
- 认证类型是否一致
- 参数名称是否正确
- 流程步骤是否完整

结果: ✅ 正确 / ⚠️ 部分正确 / ❌ 错误
```

**维度 4: 依赖包检查**

```
从文档提取:
- 包管理器（npm/pip/maven/...）
- 包名称
- 推荐版本

查询验证:
- 包是否存在
- 最新版本号
- 安装命令是否正确

结果: ✅ 正确 / ⚠️ 版本过旧 / ❌ 包不存在
```

**维度 5: 初始化顺序**

```
从文档提取:
- 初始化步骤
- 配置要求
- 调用顺序

对比 AI 代码:
- 初始化是否在调用之前
- 配置项是否完整
- 调用顺序是否正确

结果: ✅ 正确 / ⚠️ 部分正确 / ❌ 顺序错误
```

**维度 6: 版本兼容性**

```
从文档提取:
- 当前文档版本
- API 变更历史
- 废弃/新增的 API

对比 AI 代码:
- 使用的 API 在当前版本是否可用
- 是否使用了已废弃的 API
- SDK 版本是否匹配

结果: ✅ 兼容 / ⚠️ 部分兼容 / ❌ 不兼容
```

**维度 7: 语法检查**

```
执行:
- 根据语言选择 linter
- JavaScript/TypeScript: eslint 或 node --check
- Python: python -m py_compile 或 pylint
- Go: go vet
- Java: javac -d /tmp
- 其他: 尝试语法解析

结果: ✅ 通过 / ⚠️ 警告 / ❌ 错误
```

#### 步骤 5: 生成验证报告

使用 `templates/report-template.md` 模板生成报告。

可选：运行 `validate.sh` 生成验证任务骨架：

```bash
bash skills/code-verify/validate.sh --tool "工具名" --code "代码内容" [--lang 语言] [--doc-url URL] [--output report.md]
```

## 🎯 置信度等级

| 通过数 | 置信度 | 建议 |
|--------|--------|------|
| 7/7 | 100% | ✅ 可以直接使用 |
| 6/7 | 86% | ✅  minor 问题，可安全使用 |
| 5/7 | 71% | ⚠️ 需要人工确认存疑项 |
| 4/7 | 57% | ⚠️ 建议修正后再使用 |
| 3/7 | 43% | ❌ 不建议直接使用，需大幅修改 |
| 2/7 | 29% | ❌ 方向可能错误，建议重新生成 |
| 0-1/7 | 0-14% | ❌ 完全不可信，必须重写 |

## 🔧 技术实现

### 核心工具调用

```
web_search    → 搜索官方文档
web_fetch     → 抓取文档内容
browser       → 浏览器访问（web_fetch 失败时）
exec          → 语法检查、依赖查询
```

### 语法检查命令速查

| 语言 | 命令 | 安装要求 |
|------|------|---------|
| JavaScript | `node --check file.js` | Node.js |
| TypeScript | `npx tsc --noEmit file.ts` | TypeScript |
| Python | `python3 -m py_compile file.py` | Python 3 |
| Go | `go vet file.go` | Go |
| Java | `javac -d /tmp file.java` | JDK |
| Ruby | `ruby -c file.rb` | Ruby |
| PHP | `php -l file.php` | PHP |
| Rust | `rustc --emit=metadata file.rs` | Rust |

### 依赖包查询命令

| 包管理器 | 查询命令 |
|---------|---------|
| npm | `npm view {package} version` |
| pip | `pip index versions {package}` 或 `pip search {package}` |
| Maven | 访问 `https://search.maven.org` |
| Go | `go list -m -versions {package}` |
| RubyGems | `gem search {package} --remote` |

## ⚠️ 注意事项

1. **文档时效性**: 优先使用最新文档，注意文档版本号
2. **多语言文档**: 如果官方有中文文档，优先使用中文（减少翻译误差）
3. **交叉验证**: 重要 API 在多个页面交叉验证（文档 + 示例 + release notes）
4. **不要过度依赖**: 验证报告是参考，最终决策权在用户
5. **失败降级**: 如果某个维度无法验证（如文档不可达），标记为 ⚠️ 而非 ❌
