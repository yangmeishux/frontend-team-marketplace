## 2. 《变更规格》建议区块（Change Spec）

写作时逐项填满；无信息则显式 **TBD**。

1. **切片与范围**（引用本节实操记录第一节 + 范围变更记录）  
2. **现状 as-is**（每条附 `路径 + 符号或行号`；Greenfield 模式附模式库引用）  
3. **目标 to-be**（可验证；证据不足时写「待 Owner 收紧」）  
4. **不变量**（行为 + 证据或 TBD）  
5. **依赖与因果**（含待验证）  
6. **接口与数据**（请求/响应/空值；字段矩阵链到实操记录）  
7. **风险与回滚**  
8. **与影响面清单对账**

| 清单条目 | 在本规格中位置 | 状态 |
|----------|----------------|------|
| | | |

## 4. 实现后核对 · 差异表（步骤 4）

| 文档原述 | as-built（代码证据） | 建议（改文档 / 改代码 / TBD 问后端） |
|----------|-------------------------|--------------------------------------|
| | `路径:L___` | |

**默认**：未授权不改业务代码；冲突以代码为 as-built 除非用户要求以文档为准落地代码。

## 5. 源文档版本演进 · 对齐分析骨架（Reconcile R1 产出）

落盘建议：`<规格根>/<新版本号>-对齐分析.md`（文件名可本地化，保留版本号）。

```markdown
# <新版本号> 对齐分析

| 字段 | 值 |
|------|-----|
| 当前版本 | vN |
| 上一版本 | vN-1 |
| 比对人 | |
| 比对日期 | |

## 一、应剔除（vN 已删除 / 收窄，但 vN-1 曾误加）
| 项 | vN-1 处置 | vN 现状 | 剔除范围（实操记录/Change Spec/契约/代码） | 责任产物路径 |
|----|-----------|---------|---------------------------------------------|--------------|

## 二、应新增（vN 新增）
| 项 | vN 出处 | 影响文件 | 状态 |

## 三、仍保留（两版本一致）
- …

## 四、未决（待 Owner 裁决）
- 提问 / 阻塞点 / 问谁

## 五、同步任务（R2~R5）
- [ ] R2 实操记录 §一 范围变更记录已更新
- [ ] R3 增量勘探已完成（或显式沿用 vN-1 勘探 + 增量）
- [ ] R4 Change Spec 受影响段已重写、无 vN-1 残段
- [ ] R5 下游产物（若涉及）已同步、版本号一致
```
