# 变更规格全流程技能

## TL;DR（Agent 先读）

1. **缺规格根目录或未读到任何规格文件** → 只输出「待补充清单」，**不**写实操记录正文、**不**写 Change Spec。  
2. **步骤 1→2→3 必须按序**；步骤 2→3 **闸门**不满足则 **to-be 保守**。步骤 4 须用户声明开发完成；**默认 as-built=代码、只改文档**。  
3. **落盘结构**对齐同目录 **[reference.md](reference.md)**；首次写《实操记录》/Change Spec 时**应对照**其中骨架，减少格式漂移。  
4. **子步骤**只跑 **§一** 所列一步；**闸门未过**须在 **§四** to-be 贴套话；**达标打分**见 **§九**（须结合 `test-prompts.json` 实测）。  
5. **源文档版本号变化**或用户说「按 vN 重审 / reconcile」→ 走 **§四步骤 1 之 Reconcile 子模式**（R1 对齐分析 → R2~R5 同步），**不**从零重跑全部步骤、**不**留旧版残段。

本技能描述一套**可迁移的团队流程**：先划清本期范围，再弄清代码影响面，再把可执行事实落成变更规格，最后在需要时做**文档与实现对账**。文件名以团队惯例为准。

## 与 Darwin.skill 式迭代的衔接（可选）

本仓库可配套 **[test-prompts.json](test-prompts.json)**：供「带技能 vs 基线」对比打分（参见 [darwin-skill](https://github.com/alchaincyf/darwin-skill) 的 **8 维 rubric** 与 **棘轮保留**）。**不安装** Darwin 也可正常使用本技能；安装后可用 `npx skills add alchaincyf/darwin-skill` 做自动化 hill-climbing，**每次实验只改本 SKILL.md**，改后须 **独立评测**（子 Agent 或人工）再决定是否保留。实测记分见 **§九** 与 **[results.tsv](results.tsv)**。

## Greenfield 模式（新功能 / 无现有代码）

当本期需求为 **全新功能**（仓库中无对应 as-is 代码）时，自动启用 Greenfield 模式，修改以下规则：

| 常规模式 | Greenfield 模式 |
|----------|----------------|
| as-is 引用 **现有代码** 路径+符号/行号 | as-is 引用 **模式库**（同类页面/列表/配置/导出/表单等已有实现模式），格式：`参考 <同类页面路径> · <组件/模式名> · 约 L___–___` |
| 闸门要求 as-is 多数含路径+行号 | 闸门判定 **独立**：as-is 列 **参考的模式文件**（至少 1 个）+ to-be 描述 **新建路径**（`src/modules/...` 草案）即可过闸 |
| 闸门未过时贴「证据不足」套话 | Greenfield 下 as-is 写模式引用即视为 **证据充足**，无需贴套话；但若 **连模式库也找不到**（零同类页面），则套话仍适用 |
| 不变量从现有行为推导 | 不变量从 **模式库行为 + PRD 约束** 推导，标注「推导」 |

**启用条件**（满足其一）：
- 用户首条声明「新功能 / Greenfield / 从零开发」
- 步骤 2 搜索 PRD 关键词 **零命中** 且用户确认无等价旧功能

**Greenfield as-is 示例**：
```markdown
- 参考 `src/modules/<同类模块>/` 列表页模式（分页+筛选+操作列），约 L1–120
- 参考 `src/modules/<已有模块>/` 配置导出模式，约 L45–80
- 新建路径草案：`src/modules/<新模块>/`（列表 + 状态切换 + 批量操作）
```

## 与其它技能的分工（避免抢活）

- **`openspec-contract-authoring`**（或团队等价）：OpenSpec **四文件、字段矩阵、闸门句式** 规范化。  
- **本技能**：**步骤顺序**、勘探证据、Change Spec + 实操记录落盘、可选实现后核对；**不**展开「矩阵逐字段怎么写」——可并行 `@` 契约技能。

## 一、固定执行顺序与 I/O

默认**同一轮任务**内**严格按序**；用户首条写「只执行步骤 N」则**只做 N**（**子步骤模式**）：

- **禁止串步**：不得在本轮撰写其它步骤的**完整专用产出**（例：仅步骤 2 → **不**新建/覆盖 Change Spec；仅步骤 3 → **须有**已存在的步骤 1～2 成果或同目录已有《实操记录》再写，否则先列**缺哪段材料**）。仅步骤 4 → 须有可读源码 + 既有 Change Spec/规格，否则先列缺口。  
- **仅步骤 1**：只产出**切片**（实操记录第一节或等价）；**不**展开候选文件表 / 主链路长表、**不**写 Change Spec。  
- **仅步骤 2** 且无现成「本期 In/Out」：可从**已读规格**归纳一段 **In/Out（草案）** 并标注「待用户确认」再开搜；**归纳不能**时只输出待补充清单。  
- **仅步骤 3**：在 Change Spec 开篇用 **5 行内**列出「将引用的实操记录路径、闸门是否满足、若不满足已贴 §四 套话」；**不得**跳过闸门声明。缺少步骤 1～2 材料时只列缺口。  
- **仅步骤 4**：只产出 **reference.md §4** 式差异表 + 文档修订建议；默认不改业务代码。  
- **收尾 §七 仍适用**（路径列表可只列本步触及的文件/段落，或「本轮未落盘」）。

**超大仓 / monorepo**：首条建议含 **In scope**（模块、路径前缀或包名）；步骤 2 **仅**在此范围搜索，外圈一律 **TBD（未勘探）**。若用户未给范围但明确要求继续步骤 **1→2→3**，不要卡死：先按已读规格中的模块名、路由、关键词在可读范围做保守勘探，步骤 2 标题下写明「全库未承诺全覆盖，外围 TBD」，不得假装已全库覆盖。

| 步骤 | 名称 | 依赖的上游输出 | 本步完成后须有（输出） |
|------|------|----------------|------------------------|
| 1 | 需求切片 | 可读到的规格文件 | 《实操记录》**第一节**（或等价）：业务一句话、In/Out、验收、依赖、TBD、防漂移风险 |
| 2 | 影响面勘探 | 步骤 1 的 In/Out | 实操记录续：候选文件表、主链路、谁写谁读；可选字段矩阵 |
| 3 | Change Spec | 步骤 1～2 | `变更规格-Change-Spec.md`（或与 proposal 同级单文件）：范围、as-is、to-be、不变量、依赖、接口与数据、风险、**对账表** |
| 4 | 实现后核对（可选） | 步骤 3 + **用户声明开发完成** | 差异表 + 文档修订建议；默认**不改**业务代码 |

| 顺序栏（总览） | 做什么 | 典型产出名（可替换） |
|----------------|--------|----------------------|
| 1 | 切片：收敛本期、防任务/spec 漂移 | 实操记录 · 切片章 |
| 2 | 勘探：证据检索与主链路 | 实操记录 · 表与清单 |
| 3 | 变更规格：可执行真相 + TBD | `变更规格-Change-Spec.md` |
| 4 | 核对：文档 vs 代码 | 修订清单 / 勾选表 |

步骤 3 依赖步骤 2 的**闸门**（见第四节）；步骤 4 依赖可读源码与既有规格文件。

## 二、开工前输入契约（缺则停）

进入步骤 1 前核对；**不满足则只输出「待用户补充」bullet 列表**，**禁止**虚构《实操记录》或 Change Spec 正文：

1. **规格根目录**：已 `@` 或可访问，且能 `Read` 到**至少一份**规格材料（`proposal`/`design`/`tasks`/spec 等以**实际存在**为准）。  
2. **巨仓**：建议有 **勘探范围**；若用户未给或拒绝提供但要求继续，步骤 2 标题下写明「全库未承诺全覆盖，外围 TBD」再继续；若连可读规格关键词也无法归纳，则按缺输入列清单。

**工具不可读仓库**：说明限制，列出需用户粘贴的路径/片段；**停止**编造文件内容。

**源文档在工作区外**（如用户提供绝对路径 / 桌面 / 下载目录 / 外部链接）：

- 源文档**可读但无法 `@` 进 repo** 时，仍可继续执行；但须在步骤 1 开头输出提示：「⚠️ 源文档在工作区外，建议复制到 `<规格根>/<约定子目录>/<文件名>-vN.md` 以便后续 `@` 引用与 diff」
- 该风险须 **登记为 TBD**（"源文档 vN 未入库"）并随《实操记录》§TBD 流转
- **禁止宣称"已按 vN 定稿"**，除非源文档已 copy 进仓库或团队书面允许例外（须在 Change Spec 顶部记录例外）

**子步骤与契约**：N∈{1,2,3} 时 **§二 之 1（规格根目录）**仍必须；N=4 时另须 **可读源码** + **既有 Change Spec 或等效规格段落**，否则按「缺则停」列清单。

## 三、如何触发本技能

`disable-model-invocation: true` → 须**显式**触发。

1. **@ 引用本技能**：安装插件后 `@` **`skills/change-spec-workflow/SKILL.md`**（以 Cursor 实际展示为准）；或复制到 `~/.cursor/skills/` 后 @ 该路径。  
2. 文中含 **`change-spec-workflow`** 或 **「按变更规格全流程」** + @ **规格根目录**  
3. **同义**：一次性切片+勘探+Change Spec、联调前补规格、开发完成文档反扫  
4. **子步骤**：首条「只执行步骤 N」，N∈{1,2,3,4}

可选：团队长篇流程文档自愿 `@`；本技能**不绑定**固定路径。

## 四、各步骤要点与闸门

### 步骤 1 — 需求切片

- 只读**实际存在**的规格文件。  
- 输出：业务一句话、In/Out、验收、依赖、TBD（问谁）、防漂移风险。

**多文档优先级**（当源文档含多个文档/版本时）：
1. **正式版源文档**（带版本号/日期）为基准
2. **补充说明 / 新需求说明** 为 diff，冲突处显式列出
3. **原型 / 交互稿** 仅补交互细节
- 冲突条目须列 **Open Questions**，不得自行合并

**源文档解读规则（防误读）**：源文档常在不同小节描述 **不同端 / 不同角色 / 不同对象** 的能力；切片时须对每条能力**显式归属**到 `<端 × 角色 × 对象>` 三元组，再纳入 In/Out。

- **跨端 / 跨角色 / 跨对象不得合并**：不同入口（如管理后台、移动端、第三方端）的能力即使描述相邻，也须分行登记；归属不明先标 **Open Question**，禁止"就近合并"。
- **模糊措辞强制转 Open Question**：源文档出现「**同一套 X / 共用 / 复用 / 类似 / 与 Y 一致**」等措辞时，不得自行解读为"共享实现"或"共享契约"。须原文摘录 + 列入 Open Questions，由 Owner 裁决"复用何物（组件 / 接口 / 字段 / 配置）"。
- **新旧轨道识别**：源文档若同时出现新旧字段、新旧 token、新旧术语（带「（旧）」「废弃」「兼容」字样或版本对照表），切片须输出 **新旧轨道对照表**，并显式标注每一项的本期归属（**新轨纳入 / 旧轨静默兼容 / 旧轨整体下线**）；任一项未裁决前禁止写入 to-be。

**新旧轨道对照表**（示例表头，按需扩展列）：

```markdown
| 维度 | 新轨（本期） | 旧轨（兼容/下线） | 本期处置 | 证据/出处 |
|------|--------------|-------------------|----------|-----------|
| | | | 新轨纳入 / 旧轨静默兼容 / 旧轨下线 / TBD | 源文档 §x.y.z |
```

**源文档版本演进 · Reconcile 子模式**（源文档发布 **新版本号 / 修订** 时触发）：

当源文档**版本号变化**、或用户显式说「按 v___ 重新对齐 / 重审 / reconcile」时，须按下表执行 **重审**，而不是从零重跑全部步骤：

| Reconcile 子步骤 | 必做 | 产出 |
|-----------------|------|------|
| R1 · diff 源文档 | ✅ | `<规格根>/<新版本号>-对齐分析.md`：**已剔除**（旧版本曾误加、新版本删除/收窄）、**需新增**（新版本新增）、**仍保留**（两版本一致）三段表 |
| R2 · 重审切片 | ✅ | 在《实操记录》§一新增「范围变更记录」一行（提出者=源文档 vN）；更新 In/Out |
| R3 · 增量勘探 | ⚠️ 仅 R1 涉及的新增 / 收窄部分 | 旧勘探可沿用，但须在表头标「沿用 vN-1 勘探 + vN 增量」 |
| R4 · Change Spec 同步 | ✅ | 重写 to-be 受影响段；**禁止留旧版残段** |
| R5 · 下游同步 | ✅ | 触发本插件 / 兄弟技能的"局部刷新护栏"（若编排技能存在，由其约束阶段 B / 实现代码同步） |

**R1 对齐分析模板**（落盘文件骨架，**版本号 / 字段名 / 模块名等占位** 须替换为本期真实值）：

```markdown
# <新版本号> 对齐分析

## 一、应剔除（vN 已删除 / 收窄，但 vN-1 曾误加）
| 项 | vN-1 处置 | vN 现状 | 剔除范围（实操记录/Change Spec/契约/代码） |
|----|-----------|---------|------------------------------------------|

## 二、应新增（vN 新增）
| 项 | vN 出处 | 影响文件 |

## 三、仍保留（两版本一致）
- …

## 四、未决（待 Owner 裁决）
- …
```

**Reconcile 完成的判据**：实操记录、Change Spec、（若编排到下游）四文件契约、实现代码 **均不再含 vN-1 误加 / vN 已删除项**；R1 表三栏与各产物相互核对。

---

**范围变更记录**（用户在对话中收窄/扩大范围时，显式写 delta）：

```markdown
### 范围变更记录
| 日期 | 原始范围 | 变更后 | 提出者 | 影响 |
|------|----------|--------|--------|------|
| | | | | |
```

- 用户收窄范围（如「本期只做模块 A，模块 B 待定」）时，**立即**在实操记录中更新 In/Out 并记录变更原因
- 范围变更后，步骤 2 勘探范围同步收窄

**change-id 确认停顿**（步骤 1 产出后 **强制停顿**）：
- 步骤 1 末尾须输出：「`change-id` 建议为 `<xxx>`，请确认后再进入步骤 2」
- **未获用户确认前不得进入步骤 2 落盘**（用户明确说「不用确认」或「直接继续」除外）  

### 步骤 2 — 影响面勘探

- 检索词 → 在**约定范围**内搜索。  
- 产出：候选文件表、主链路（路径→符号→约行号）、谁写谁读；矛盾 → **TBD**。  
- **首次**在实操记录中落长表前，应对照 **reference.md §1** 骨架（节名可改，**板块勿缺**）。

### 步骤 2 → 3 闸门（写满 to-be 前）

满足**之一**才可写**进取型 to-be**；否则 **to-be 保守**并首段声明「证据不足，待 Owner 收紧」：

- 与 In scope 相关的 as-is **多数**含 **路径+符号/行号**；或  
- 用户首条写明「接受证据不足草案」。

**资金/合规/核心交易主链路**无证据：**禁止**在 to-be 写死「已上线/已实现」类结论。

**人在回路（建议）**：若 to-be 将驱动**对外承诺或上线决策**，在步骤 3 定稿前用 **一小段「须 Owner 确认」** 列出 2～3 条，不代替人类拍板。

**闸门未过 · to-be 段首套话（须贴在 Change Spec「to-be」小节最前；表述可本地化，**五层含义须保留**：草案性质、闸门未满足、待 Owner 收紧、禁止对外承诺、核心路径见 TBD）：

```markdown
> **草案性质（证据不足）**：步骤 2→3 **闸门未满足**——与本期 In scope 相关的 as-is **尚未多数**具备「路径 + 符号/行号」，且用户**未**声明「接受证据不足草案」。本节 to-be 仅为**方向性草案**，须在 Owner 补齐证据后收紧；**不得**据此对外承诺或写死「已上线 / 已实现」。资金/合规/核心交易主链路无证据处 **仅**能写假设或 TBD，详见文末对账表。
```

### 步骤 3 — Change Spec

- 区块：**范围、as-is（带来源）、to-be、不变量、依赖与因果、接口与数据、风险与回滚、对账表**。结构与 **reference.md §2** 对齐。  
- **不得**删 TBD；**不得**改 `proposal`/`design`**除非**用户明确要求。

**Change Spec 必填块 ↔ reference.md §2（落盘时逐条有标题；无信息写 TBD）**：

| §2 项 | 文中须能检索到的关键词（团队可改名，**语义须对应**） |
|-------|--------------------------------------------------------|
| 1 切片与范围 | 本期范围 / In scope / 与实操记录切片对齐 |
| 2 现状 as-is | as-is / 现状；条目标 **路径+符号或行号** |
| 3 目标 to-be | to-be / 目标；**闸门未过则 §四 套话居首** |
| 4 不变量 | 不变量 / 底线行为 |
| 5 依赖与因果 | 依赖 / 因果 / 上下游 |
| 6 接口与数据 | 接口 / 数据 / 字段；矩阵见 reference §3 |
| 7 风险与回滚 | 风险 / 回滚 |
| 8 对账 | 与影响面清单对账 **表**（含清单条目与状态） |

### 步骤 4 — 实现后文档核对

- 须用户明确「开发完成/即将合入」。  
- 冲突且未说「以文档为准改代码」→ **as-built = 当前代码**；规格侧「待决议」。  
- **默认只改文档**；**禁止**为对齐叙述**重构业务代码**；用户书面授权改代码时除外。

**人在回路**：若建议**改代码**，必须先列出 diff 意图并**停下**，待用户确认（本技能默认不自动合入代码改动）。

## 五、异常与降级

| 场景 | 处理 |
|------|------|
| 搜索零命中 / 超时 | 写 TBD + 建议「缩小关键词 / 换入口文件」；不编造命中路径 |
| 规格目录可列目录但某文件编码失败 | 记录文件名 + TBD，不脑补内容 |
| 步骤 3 与用户已有多份 Change Spec | 问用户「覆盖 / 另存 `-v2`」；默认**不覆盖**未点名文件 |
| 步骤 4 与步骤 3 结论严重冲突 | 差异表列出；**不自作主张**选边，标「须 Owner/后端裁决」 |
| token/回合将尽 | **优先**完成当前步骤 + **§七** 三节小结；建议下一轮从步骤 N 继续 |
| 用户指令矛盾（例：既要 1→2→3 又写「只步骤 2」） | **停下**问清优先级；未答复前按**较窄**范围执行（通常以**后写**的「只步骤 N」为准） |

## 六、Agent 自检（结束前）

- [ ] 输入契约曾被违反却仍写满篇？  
- [ ] 用户指定**只步骤 N** 时是否**未串步**（尤其「仅 2 却写满 Change Spec」）？  
- [ ] 闸门与人在回路（to-be / 改代码）是否遵守？  
- [ ] as-is 是否附 **路径+符号/行号**？TBD 是否未闭合？  
- [ ] 闸门未过时，Change Spec **to-be 段首**是否已贴套话或**等价**声明（非悄悄写满进取型 to-be）？  
- [ ] Change Spec 是否覆盖 **§四「步骤 3」下 8 行对照表**的语义（可改名，缺项须 TBD）？  
- [ ] 源文档**跨端 / 跨角色 / 跨对象** 能力是否**显式归属**、未"就近合并"？  
- [ ] 源文档**模糊措辞**（同一套 / 共用 / 复用 / 类似）是否进 Open Questions、未自行解读？  
- [ ] **新旧轨道**是否已列对照表、每项均有「新轨纳入 / 旧轨静默兼容 / 旧轨下线 / TBD」裁决？  
- [ ] 源文档若发布**新版本**，是否走了 Reconcile 子模式（R1 对齐分析 → R2~R5 同步），未留 vN-1 误加 / vN 已删除残段？  
- [ ] 源文档在工作区外时，是否提示复制入库、并将"未入库"登记为 TBD？  
- [ ] 参考资料 reference.md / test-prompts 预期是否被无理偏离（自评，dry_run）？  
- [ ] §七 **三节是否齐全**、是否曾把三块信息**挤成一段**？

## 七、收尾输出格式（强制）

任务结束前须包含下面 **三节**（标题可本地化，**三块信息不得合并成一段**）：

1. **本轮写入或修改的路径**（列表；注明新建/覆盖/追加章节）  
2. **仍为 TBD**（每项 **问谁** / 问哪团队）  
3. **建议下一步**（**可执行**的一项或短列表：如联调、补矩阵、仅步骤 4、Owner 审）

**结构示例**（占位须换成本轮事实；若仅对话未 `Write` 任何文件，须在第一节写明「本轮未落盘」）：

```markdown
### 本轮写入或修改
- `specs/foo/实操记录.md`（新增 §一～§三）
- `specs/foo/变更规格-Change-Spec.md`（新建）

### 仍为 TBD
- 下游服务 Z 是否验收新字段（问：@后端某某）
- …

### 建议下一步
- Owner 确认 Change Spec §to-be 两条表述后合入；或下一轮只跑步骤 4
```

## 九、质量评分（达标线 **≥90 / 100**）

与 [darwin-skill](https://github.com/alchaincyf/darwin-skill) 同类：**结构可静态复核 + 效果须独立会话实测**。满分 **100 = 结构 60 + 效果 40**。

### 9.1 结构分（0–60）

以下 **12** 条在评审阅读本 SKILL 时核查，**每条 5 分**（满足得 5，明显缺失得 0，边界情况由评审裁量 1～4）：

| # | 核查点 |
|---|--------|
| S1 | Frontmatter：`name`、`description`（触发/边界/分工）、`disable-model-invocation` |
| S2 | TL;DR 覆盖：缺输入、顺序+闸门、落盘、子步骤/打分入口 |
| S3 | §一：**I/O 表** + 子步骤 1/2/3/4 **不串步** |
| S4 | §二：输入契约 **缺则停** + 子步骤与规格根/源码要求 |
| S5 | §三：显式触发方式 |

### 9.2 效果分（0–40）

对 **test-prompts.json** 中 **id 1～6**，**各在单独一次**带本技能的 Agent 会话中执行，按该条 `expected` **二元判据**记 `pass` / `fail`：

`效果分 = round( (pass 条数 / 6) × 40 )`（**6/6 ⇒ 40**；**5/6 ⇒ 33**）。

将结果写入同目录 **[results.tsv](results.tsv)**：`prompt_id`、`pass`（1/0）、`date`、`model_or_reviewer`、`notes`。

### 9.3 总分与「≥90」认证

**总分 = 结构分（§9.1）+ 效果分（§9.2）**，各不超过 60 / 40。

- **宣称技能版本 ≥90**：须 **结构 + 效果 ≥ 90**。  
- **高置信（推荐对外）**：**效果 = 40**（**test-prompts 6/6**）**且** **结构 ≥ 55**。  
- **仅静态读 SKILL、未跑 prompts**：不宣称 ≥90；可写「结构上界约 **55～60**（自评）」。

**本版自我期许**：正文可对齐 §9.1 全部核查点；**你在本机跑通 `test-prompts` id1～6 并记入 `results.tsv` 后**即可按上式得到认证总分。
