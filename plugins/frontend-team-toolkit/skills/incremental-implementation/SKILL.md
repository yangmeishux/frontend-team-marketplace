# Incremental Implementation — 锚定现有代码，再做增量迭代

> **版本**: 1.1.0
> **定位**: 把「开发文档的 delta」落成「旧代码上的最小 diff」的可执行技能（非代码生成器，非从零脚手架）
> **边界**: 只做增量实现与回写映射；产品取舍、契约定稿见 `pm-md-to-openspec-pipeline` / `openspec-contract-authoring`

---

## 一、铁律（Agent 先读）

**已经实现过的需求，再次迭代时，唯一事实源是「已提交的代码」，不是开发文档。**

LLM「从零生成代码」天然不可复现——同一份文档、同一个模型，命名 / 拆分 / 文件组织都会自由发挥，必然漂移。
所以本技能的核心不是「让全量重跑变稳定」，而是**根本不全量重跑**：

1. **禁止 greenfield 重建**：已存在的文件 / 组件 / 函数，不得整体重写或另起新文件覆盖。
2. **只生成 delta**：每一项变更必须落为「在现有代码上的最小编辑」（新增 / 修改 / 删除），未变项**显式跳过并断言**。
3. **git 基线先行**：动手改代码前，必须有干净的 git 基线（commit 或分支），改完用 `git diff` 自检。
4. **代码 ↔ 任务 双向可追溯**：通过 `实现映射.md` 把每个任务锚定到具体文件 / 函数；没有锚点就先补锚点。
5. **源文档版本演进走 Reconcile 代码编排**：上游 `<vN>-对齐分析.md` 三栏是变更分类首选输入；缺对齐分析时**不臆造分类**，见 §五。
6. **多产物版本号一致**：`实现映射.md`、`tasks.md`、契约四文件、上游 Change Spec 须共享同一 `基准源文档版本：vN`；漂移时先对齐再改代码。
7. **局部代码更新须护栏**：用户只改部分模块 / 文件时，输出「未同步产物清单 + 风险声明」，禁止悄悄宣称 R7 已闭合。

> 一句话：**哪怕开发文档是全新生成的，执行阶段也被「现有代码 + git + 实现映射」死死锚住，产物必然是旧代码上的二次迭代，而非从零开始。**

**迭代 Backlog**：各项目执行反馈见 **[BACKLOG.md](BACKLOG.md)**。

## 二、适用 / 不适用

| ✅ 适用 | ❌ 不适用（走其它技能或直接开发） |
|--------|-----------------------------------|
| 需求已实现过，PRD 升版本后要改 | 全新项目首次实现（走正常开发，本技能无旧代码可锚） |
| 需求二次 / 多次迭代 | 纯文档编写（走 `pm-md-to-openspec-pipeline`） |
| Reconcile 已产出对齐分析，要落地代码 | 与 openspec/changes 无关的临时重构 / 样式微调 |
| 开发文档重新生成了，但要在旧代码上改 | — |
| 源文档 vN 升版后的 **R6 代码同步** / **按 vN 全链同步代码** | — |
| 用户要求「只改某模块 / 某 API / 某页面」的局部代码更新 | — |

**前置条件（缺则停，先补齐）**：
- 仓库是 git 仓库且**工作区干净**（无未提交改动，否则基线不可信）；
- 存在已实现的代码（首版已落地）；
- 存在变更依据，**按优先级**：
  1. **Reconcile 对齐分析** `<vN>-对齐分析.md` 三栏（应剔除 / 应新增 / 仍保留）—— **首选**；
  2. 新旧 `tasks.md` + `field-matrix.md` 的差异（须标注「契约未 reconcile 风险」）；
  3. 用户书面列出的剔除 / 新增清单（须回链到源文档 § 出处）。

**缺对齐分析时的处置**：
- 用户说「按 vN 改代码」但无 `<vN>-对齐分析.md` → **停下**，建议先跑 [`../pm-md-to-openspec-pipeline/SKILL.md`](../pm-md-to-openspec-pipeline/SKILL.md) Reconcile R0~R5；或用户明确「仅凭 tasks diff 改代码，接受契约漂移风险」才继续。

## 三、核心机制：`实现映射.md`（任务 ↔ 代码 锚点）

裸跑 `tasks.md` 漂移的根因：任务只有 `[ ]/[x]`，**不知道某任务到底落在了哪个文件**，重跑时只能重新发挥。

解法：维护一份**实现映射**，把每个任务锚定到代码。它是「代码即事实源」的载体。

- 落盘路径（与规格同目录）：`specs/<change-id>/实现映射.md`
- 模板与字段见 **[reference.md §1](reference.md)**
- 每条至少含：任务 ID、状态、**实现锚点（文件 / 函数 / 关键符号）**、最近变更 commit、关联契约项

> 首次进入本技能时若 `实现映射.md` 不存在：**先做一次「反向建图」**（P0.5），从现有代码回填映射，再开始增量。

## 四、五阶段执行流水线（必须按序）

```text
P0  基线锚定      git 基线 + 读实现映射 + 读变更依据
      ↓
P0.5 反向建图     （仅当映射缺失）从现有代码回填 实现映射.md
      ↓
P1  变更分类      对照对齐分析 → 每项归入【新增/修改/删除/不变】
      ↓
P2  影响定位      每个【新增/修改/删除】项 → 定位现有文件/函数/调用点
      ↓
P3  最小编辑      三态落地 + 【不变】项显式跳过断言；禁止重建
      ↓
P4  diff 自检     git diff 复核 + 回写 实现映射.md 与 tasks.md 状态
```

### P0 · 基线锚定

1. `git status` 确认工作区干净；不干净则**停**，请用户先提交 / 暂存。
2. 建基线：`git commit`（空跑则跳过）或新建迭代分支，记录基线 commit 短哈希。
3. `Read` `实现映射.md`（无 → 转 P0.5）、`tasks.md`、`<vN>-对齐分析.md`（或新旧 tasks 差异）。

### P0.5 · 反向建图（仅当映射缺失）

逐条扫已实现的 `tasks.md`，到代码里定位实现位置，回填 `实现映射.md`：
- 用 Grep / 语义检索定位每个任务对应的文件与关键符号；
- 不确定的标 `锚点待确认`，不要臆造；
- 建完图再进入 P1。

### P1 · 变更分类（核心一步）

把本轮所有变更逐项归入四类，输出**变更分类表**（模板见 reference §2）：

| 类别 | 来源 | 处置 |
|------|------|------|
| **新增** | 对齐分析「应新增」/ 新 tasks 多出的项 | 新功能：可新建文件，但须先查是否有可复用现存模块 |
| **修改** | 对齐分析「仍保留」但行为/字段变了 | **定位现有实现 → 原地最小编辑** |
| **删除** | 对齐分析「应剔除」 | 定点删除代码 + 清理引用 |
| **不变** | 对齐分析「仍保留」且无变化 | **显式跳过，断言「已实现于 X，本轮不动」** |

> 「不变」项必须显式列出并断言不动——这是防止 AI 顺手重写旧代码的关键护栏。

### P2 · 影响定位

对每个【新增/修改/删除】项：
- 用 `实现映射.md` + Grep + 语义检索，列出**受影响文件 / 函数 / 调用点 / 测试**；
- 「修改」「删除」必须先找到现存落点，找不到 → 退回 P1 重新归类（可能其实是新增）；
- 输出**影响定位表**（reference §3）。

### P3 · 最小编辑执行

- **修改**：在现有文件用 StrReplace 做最小补丁，保留原有结构 / 命名 / 风格；**禁止**整文件重写。
- **新增**：优先复用现存组件 / composable / api 模块；确需新建时遵循仓库目录约定（见 AGENTS.md「开发约定」）。
- **删除**：删实现 + 删引用 + 删死代码，避免悬挂导入。
- **不变**：一行不碰；若工具触碰到，回退。
- 每完成一项即可小步提交（commit 信息中文，见全局规则）。

### P4 · diff 自检 + 回写

1. `git diff <基线>..HEAD`（或 `git diff`）逐文件复核：是否只动了该动的、有无误改「不变」项。
2. 回写 `实现映射.md`：更新被改任务的锚点 + 最新 commit。
3. 回写 `tasks.md`：状态勾选 + （可选）追加「→ 实现：见 实现映射.md」。
4. 跑 `code-verify` 关心的最小验证（lint / build / 关键路径），见仓库 Evidence 约定。

## 五、子阶段模式（用户首条指令）

| 模式 | 行为 |
|------|------|
| **默认** | Reconcile 代码编排 IC-R0~IC-R7（含 P0~P4） |
| **仅 P0.5 反向建图** | 只从现有代码回填 `实现映射.md`，不改业务代码 |
| **仅变更分类** | 只输出 P1 变更分类表 + P2 影响定位表，不落代码 |
| **Reconcile 代码同步** / 「按 vN 改代码」 | IC-R0~IC-R7；须可读 `<vN>-对齐分析.md` |
| **局部代码更新** / 「只改某模块」 | 启用 **局部代码更新护栏**；输出未同步产物清单 |
| **接力 pipeline R6** | 消费上游 Reconcile R0~R5 产物 + 其 R6 diff 建议，执行 IC-R3~IC-R7 |

用户指令矛盾时 **停下**问清；未答复前按 **较窄** 范围（通常「仅变更分类」或「仅反向建图」）。

## 六、Reconcile 代码编排（源文档版本演进 · 代码侧）

**触发**：满足任一项：

- 源文档版本号 / 修订日期变化（vN-1 → vN），且用户要改代码
- 用户说「按 vN 改代码 / reconcile 落地 / 全链同步代码 / 剔除误加 / 新旧不能并存」
- 上游 `pm-md-to-openspec-pipeline` Reconcile R0~R5 已完成，进入 R6

**编排顺序（不可乱序）**：

```text
IC-R0  锁定基准 vN + git 基线（工作区须干净）
         ↓
IC-R1  Read <vN>-对齐分析.md 三栏（缺则停或用户书面接受 tasks-diff 风险）
         ↓
IC-R2  多产物版本漂移检测（实现映射 / tasks / 契约四文件 / Change Spec）
         ↓
IC-R3  P1 变更分类 → P2 影响定位（模板见 reference §2~§3）
         ↓
IC-R4  P3 最小编辑（应剔除→删除；应新增→新增；仍保留→修改或不变）
         ↓
IC-R5  P4 diff 自检 + 回写实现映射 / tasks 状态 / 版本标识
         ↓
IC-R6  契约 vs 代码 diff（field-matrix 字段集 vs 实现侧；不一致→Open Question 或回 pipeline）
         ↓
IC-R7  代码 Reconcile 闸门：代码侧无 vN-1 误加项、已纳 vN 新增项、版本号一致
```

**Reconcile 子模式选择**：

| 用户口令 | 执行集 | 不做 |
|---------|--------|------|
| 「按 vN 改代码」（默认） | IC-R0~IC-R5 + IC-R7 | IC-R6（仅列 diff 建议） |
| 「按 vN 全链同步代码」 | IC-R0~IC-R7 | — |
| 「只出变更分类」 | IC-R0~IC-R3 | IC-R4~IC-R7 |
| 「接力 R6」 | IC-R1~IC-R7（跳过 IC-R0 若 git 基线已给） | 不重跑上游 R0~R5 文档 |

**禁止**：

- 跳 IC-R1 直接按新 tasks 全量实现（greenfield 漂移）
- IC-R7 未闭合就宣称「已按 vN 对齐代码 / 已实现 vN」
- 只删代码不同步 `实现映射.md` 中对应锚点（标 `已废弃`）

**IC-R7 代码 Reconcile 闸门核对清单**：

- [ ] 对齐分析「应剔除」项在代码中**已无**实现与引用
- [ ] 对齐分析「应新增」项在代码中**已有**实现锚点（或标 TBD + 问谁）
- [ ] 「仍保留 / 不变」项经 P4 `git diff` 确认**未被误改**
- [ ] `实现映射.md` 头部 `基准源文档版本：vN` 与 tasks / 契约四文件一致
- [ ] 若 IC-R6 发现 matrix vs 代码不一致：已列 Open Question 或建议回 pipeline，**未**静默忽略

细则与口令示例见 **[reference.md §6~§8](reference.md)**。

## 七、多产物版本对齐（代码 ↔ 文档 ↔ 映射）

当仓库同时存在 `specs/<change-id>/`、`openspec/changes/<change-id>/` 与源码时，本技能须**强制多产物共享同一版本标识**。

**版本标识规范**（任一项即可，团队择一）：

1. `实现映射.md`、上游 Change Spec、契约四文件头部均含 `基准源文档版本：vN`、`最近 reconcile 日期：YYYY-MM-DD`
2. `specs/<change-id>/VERSION.md` 集中列出各产物路径与基准 vN

**漂移检测**（IC-R2，进入 P3 之前必做）：

- [ ] 列出 `实现映射.md`、`tasks.md`、`openspec/changes/<id>/` 下四文件、Change Spec 的基准版本
- [ ] 不一致项标「漂移」；**禁止**在文档仍 vN-1、映射已标 vN 时继续改代码
- [ ] 修复：以**最新基准 vN** 为准，建议先跑 pipeline Reconcile R0~R5，或用户书面指定基准

**修复后**：统一更新所有受影响产物的版本标识 + `实现映射.md` 修订记录。

## 八、局部代码更新护栏（只改部分模块 / 文件）

当用户**只**要求改代码中某一个或某几个模块（如「只删某 API」「只改某页面字段」），默认启用以下护栏。

**更新前**：

1. **影响面预扫**：列出本次代码变更点 + **受牵连但未改动的产物**：
   - `实现映射.md` 对应锚点是否仍准确
   - `tasks.md` Gate / 勾选状态
   - `field-matrix.md` / `design.md` / `spec.md` 是否仍描述已删 / 已改行为
2. 输出 **「本次仅改 ＜模块/文件＞，未同步产物清单」** 表；用户明确「暂不同步文档」才继续

**必含输出**（模板见 reference §7）：

```markdown
### 本次局部代码更新
- 受改路径：…
- 变更摘要：…

### 未同步产物（漂移风险）
| 产物 | 受牵连之处 | 是否已过时 | 处置建议 |
|------|-----------|-----------|----------|

### 局部更新风险声明
> 本轮**仅**改上述代码；契约 / 映射 / tasks **未**同步至当前基准。IC-R7 **未闭合**，禁止宣称「已按 vN 对齐」。
```

**触发齐刷的判据**（满足其一）：

- 用户说「同步更新映射与契约」「按 vN 全链同步」
- 变更触及**不变量 / 接口契约 / 删除对外 API**（非纯内部重构）
- 处于 Reconcile 代码编排 IC-R4~IC-R7（Reconcile 流程内禁止局部更新后宣称闭合）

**禁止**：局部改代码后悄悄勾选 tasks Gate / IC-R7。

## 九、与其它技能的衔接

```text
源文档 vN 升版
   │
   ├─ pm-md-to-openspec-pipeline · Reconcile R0~R5   →  文档 delta（对齐分析 + 四文件刷新）
   │                              R6 仅列「代码 diff 建议」（默认不改代码）
   │
   ├─ 【本技能 incremental-implementation】           →  Reconcile 代码编排 IC-R0~IC-R7
   │     消费 <vN>-对齐分析.md + 实现映射 + git 基线
   │     产出「旧代码上的最小 diff」+ 回写映射 / tasks
   │
   └─ code-verify（可选）                             →  第三方/关键路径验证
```

- **上游**：本技能**消费** pipeline Reconcile 的 `<vN>-对齐分析.md` 三栏；文档侧变更须先 R0~R5 或用户接受漂移风险。
- **分工**：本技能**不改契约文档语义**（回 `pm-md-to-openspec-pipeline` / `openspec-contract-authoring`）；**只**把文档 delta 落成代码 diff 并回写映射。
- **下游**：`code-verify` + 仓库 `tasks.md` Evidence。

## 十、禁止项（漂移信号）

- ❌ 工作区不干净就开改（基线不可信）。
- ❌ 对「修改」类需求新建一个并行新文件，旧文件留着不管（产生双实现）。
- ❌ 整文件重写已有实现（即使"顺手优化"）。
- ❌ 跳过 P1 变更分类，直接照新 `tasks.md` 从头实现。
- ❌ 「不变」项不声明就被改动。
- ❌ 改完不更新 `实现映射.md`，导致下一轮又断链。
- ❌ 源文档 vN 变化却跳过 IC-R1 对齐分析，直接按新 tasks greenfield 实现。
- ❌ 多产物版本漂移时继续改代码（映射 vN、契约仍 vN-1）。
- ❌ 局部改代码后未输出「未同步产物清单」就宣称 IC-R7 闭合。
- ❌ 对齐分析「应剔除」项只删 UI 不删 API / 引用 / 映射锚点。

## 十一、Agent 执行自检（结束前）

- [ ] P0：工作区干净 + git 基线已记录？
- [ ] 映射缺失时是否先做 P0.5 反向建图？
- [ ] P1：输出了**变更分类表**（新增/修改/删除/不变四类齐全）？
- [ ] 「不变」项是否**逐条显式断言不动**？
- [ ] P2：每个修改/删除项都定位到了现存落点（无凭空臆造）？
- [ ] P3：修改类是否**原地最小编辑**、无整文件重写、无并行新实现？
- [ ] P4：`git diff` 已复核 + `实现映射.md` 与 `tasks.md` 已回写？
- [ ] 是否**未**在本技能里擅自改契约文档语义（该回 pipeline）？
- [ ] Reconcile 触发时是否走 **IC-R0~IC-R7**（或用户指定的子模式）？

## 十二、收尾输出格式（强制三块）

1. **本轮代码改动**：按【新增/修改/删除】分组列文件路径 + 一句变更摘要；**首行**标「**基准源文档版本：vN** · **基准 commit：`<短哈希>`** · reconcile 日期」。

若本轮未改任何代码，第一块写明「**本轮未改代码**」并说明原因（如全部归入「不变」、或缺对齐分析已停）。
