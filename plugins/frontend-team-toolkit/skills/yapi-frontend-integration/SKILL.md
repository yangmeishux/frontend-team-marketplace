# YApi 前端接口对接

## 技能边界（单一职责）

本技能只做：**通过 YApi MCP 获取接口定义 → 在前端工程中实现（及可选地推导类型）请求封装与调用方式说明**。

**不包含**（除非用户单独立项并明确范围）：

- 后端接口实现、数据库或网关配置  
- 在 YApi 上 **批量创建/修改** 接口文档（见 **`reference.md` §6**，非默认）  
- 脱离 YApi 文档的「纯猜字段」联调；契约必须来自 **MCP 拉取的文档** 或用户明示的手写契约（后者须在对话中固化后再实现）

**字段级附录、axios 映射、TS、排障**：**`reference.md`**。

---

## 执行协议（两阶段 · 方案 C）

**默认顺序**（除非用户声明 **「跳过摘要，直接改仓库」** 并自担与文档不一致风险）：

1. **第一轮交付物**（不写或仅写最小草稿）：根据 MCP 返回整理 **接口摘要**——`method`、`path`、Query/Body/Headers 要点、`res_body` 顶层结构；**拟生成**的 API 函数名、**目标文件路径**（相对业务仓库）；若需 **TS**，注明类型将来自 Schema 还是示例 JSON。  
2. **第二轮**：用户确认或小幅修订摘要后，再 **批量写入** 项目内 API 文件并接页面。

首轮**必须**先尝试拉取文档；仅在 MCP 不可用时与用户约定手写契约再继续。

## MCP 就绪（迁移前检查 · 与方案 A 衔接）

调用工具前确认：

- [ ] Cursor **Settings → MCP** 中 **`yapi-devloper-mcp`** 已启用；`YAPI_BASE_URL`、账号等与目标 YApi 一致（或本机覆盖了安全配置）。  
- [ ] 工具名以 **Cursor 内 MCP 工具描述** 为准（常见为列表类 + 详情类，参见 **`reference.md` §1**）。

---

## 主工作流（摘要）

详细步骤与字段说明见 **`reference.md` §2–§5**。

### 1. 获取接口文档

- **分类 URL** → 列表类工具 → 用户选择或缩小范围 → 详情类工具  
- **`apiId`** → 详情类工具  
- **仅接口名** → 列表筛选；**无唯一匹配则列出候选项**，禁止臆测 id

### 2. 实现 API 封装

使用项目既有 **`axios` / `fetch` 实例**；GET 用 `params`，body 用 `data`；路径参数替换占位符。可选：为 `res_body`/`req_body` 补充 **TS 类型**（**reference §4**）。

### 3. 页面调用

按文档响应结构取值；分页、错误码处理遵循项目统一模式。

## 失败分支（方案 A）

遇到以下情况时**不得**假装已完成对接：

| 情况 | 动作 |
|------|------|
| MCP 不可用或未配置 | 告知检查 MCP；可改为用户粘贴 **完整接口说明** 后再进入手写契约流程 |
| 认证失败 | 提示检查 env / 账号权限；不写依赖虚假文档的代码 |
| 列表为空或接口名多匹配 | 请用户补充 **链接 / apiId** 或从列表中指定一条 |
| 文档明显缺失 method/path | 停下并请产品/后端补 YApi，或记录 **风险** 后仅写占位 TODO |

## 交付自检（方案 A）

合并或结束任务前核对：

- [ ] 请求 **method、path、参数位置** 与 `get_api_desc`（或等价返回）一致  
- [ ] 未臆造文档中不存在的字段名（推断类型已标注依据）  
- [ ] 敏感信息未写入仓库；示例数据已脱敏
