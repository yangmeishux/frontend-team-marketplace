# Scoring Rubric — 微信公众号文章（总分 10）

## 维度与权重

| 维度 | 权重 | 9 分标准 |
|------|------|----------|
| 主题与价值 | 25% (2.5) | 主题明确、价值清晰、有独特视角 |
| 结构与逻辑 | 20% (2.0) | 结构清晰、逻辑自洽、过渡自然 |
| 干货密度 | 25% (2.5) | 干货占比 ≥60%、案例/数据有说服力 |
| 可读性与表达 | 20% (2.0) | 开头抓人、段落适中、表达准确 |
| 标题与 CTA | 10% (1.0) | 标题有吸引力、结尾有明确 CTA |

**总分** = 五维得分之和（每维满分即权重值）。

## 阈值

| 分数 | 判定 |
|------|------|
| ≥ 9.0 | 通过 |
| 8.5 – 8.9 | 接近未达标，针对性修改 |
| < 8.5 | 明显问题，重写或大幅修改 |
| 合规风险 | 直接不通过 |

## 扣分参考

| 问题 | 典型扣分 | 修改方向 |
|------|----------|----------|
| 标题平淡/夸张 | -0.3 ~ -0.5 | 数字、痛点、反差 |
| 开头 3 秒不抓人 | -0.3 ~ -0.5 | 痛点/故事/数据/问题 |
| 干货不足 | -0.5 ~ -1.0 | 案例、数据、步骤 |
| 结构混乱 | -0.4 ~ -0.8 | 重梳大纲、过渡 |
| CTA 弱/无 | -0.2 ~ -0.3 | 留言/分享引导 |
| 承诺未兑现 | 按维度 | 补小节/清单/过渡说明 |

## 承诺-正文一致性

扫描：「我会/将会/后面给你/下一节/本文提供」等。

- 未兑现 → 写入主要问题 + 最小修复（三选一：新增小节 / 补充清单 / 加过渡说明）
- 严重空头承诺 → 干货密度或结构维度重扣

## 空洞开头识别（v0.1.4 新增）

开头 3 秒判定规则：

| 开头特征 | 判定 | 扣分 |
|----------|------|------|
| 有痛点/数据/故事/问题 | 抓人 | 0 |
| 纯铺垫「XX 很重要」「本文分享」 | 空洞 | -0.3 ~ -0.5 |
| 无明确对象或价值声明 | 迷茫 | -0.2 |

**识别位置**：第 1 段（或前 3 行）

**修改建议**：
- 补痛点场景（「如果你改 Skill 只靠感觉...」）
- 补数据开头（「80% 的 Skill 没有 baseline」）
- 补问题开头（「为什么改完不知道退没退步？」）

## P0 / P1 / P2 提分路径

- **P0**：不改则无法过 9 分（结构硬伤、干货缺失、合规）
- **P1**：改后明显提分（开头、标题、案例）
- **P2**：锦上添花（措辞、可选 CTA 强化）

## 技术 Blueprint / 架构文（公众号改编）

当稿件源自仓库技术文档（如 Blueprint、SOP）时：

- **允许**保留目录树、命令块、外链表（干货维度）
- **仍必须**有结尾 CTA；缺则标题与 CTA 维度 ≤0.7/1.0
- **建议**有 150 字读者痛点场景 + 一条端到端案例（300–500 字）
- **编号**标题与列表项数量须一致（如「8 Phase」vs Phase 0–8）

### 领域专有示例（OpenSpec / 流水线 / 某框架）

- 文中出现 **领域案例**（如 OpenSpec 六条回归）**不单独扣分**，除非与主题无关
- 在 **建议（P2）** 中可写：「通用读者可替换为自己的 N 条 regression」
- **不因**「例子太垂直」给出 ❌ 不通过；仅当因此导致 **缺 CTA / 缺通用干货 / 结构硬伤** 才扣至 9 分以下

## 30 分钟提分脚本（模板）

| 时段 | 动作 | 验收句式 |
|------|------|----------|
| 0–8 min | 改标题 + 开头 1 段 | 「读者 3 秒内知道价值与对象」 |
| 8–18 min | 补 1 个案例或步骤清单 | 「至少 1 处可抄走的干货」 |
| 18–25 min | 加过渡 + 结尾 CTA | 「CTA 明确且与主题一致」 |
| 25–30 min | 通读删水句 | 「无重复段、段落 ≤3 行为主」 |
