# Output Contract — wechat-article-review

Agent 在 Workflow 步骤 2 必须 Read 本文件。所有评分报告须 Markdown 输出。

## 状态字段（报告末尾或 Summary 内）

- `DONE` — 评分完成（通过或不通过均已给清单/亮点）
- `DONE_WITH_CONCERNS` — 通过但有 1–3 条可选优化
- `BLOCKED` — 缺正文/路径；须列出最少问题集

## 通过（≥ 9.0）必含节

1. **结论**：`✅ 通过（综合评分 X.X/10）`
2. **各维度得分** — 表头含：维度 | 得分（满分按权重） | 说明
3. **加权总分计算** — 一行公式或加总说明
4. **亮点** — ≥2 条
5. **建议（可选）** — 不影响通过的优化

## 不通过（< 9.0）必含节

1. **结论**：`❌ 不通过（综合评分 X.X/10，需修改后复评）`
2. **各维度得分** — 含扣分原因列
3. **主要问题（按优先级）** — 每条含：
   - 位置（第 N 段 / 小标题「XXX」）
   - 具体表现
   - 修改建议（可执行）
   - 预期提升（约 -X.X 分回分）
4. **提分路径 P0/P1/P2** — 先做什么最划算
5. **修改清单** — `- [ ]` TODO，供文案执行
6. **复评目标** — 「改完预期 X.X–X.X 分」或说明仍不可达原因

## 条件触发

### 承诺未兑现

若文中有「将会/下文/本文提供」等未兑现：

- 输出 **承诺扣分表**：承诺句 → 位置 → 缺口 → 影响维度 → 扣分 → 最小修复 → 验收句
- 附 **复评验收清单** 2–4 条（可勾选）

### 30 分钟提分模式

用户明确时间限制时，附加：

- 0–8 / 8–18 / 18–25 / 25–30 分钟步骤
- 每步 **完成标准（验收句式）**

## Must NOT

- 不得编造未提供的段落内容或分数依据
- 不得对合规风险文章给 ≥9 分
- 不得省略不通过时的修改清单
