# frontend-dev-prompt-craft 使用指南

> 把你的前端开发需求，变成 AI 能直接执行的结构化提示词。

---

## 这个技能做什么？

你有一个前端开发需求（比如「新增一个导出页面」），但不知道怎么写提示词才能让 AI 编码工具给出高质量代码。

这个技能帮你：
1. **识别**你的需求属于哪种前端任务类型（共 8 类）
2. **追问**缺失的关键信息（路由、接口、组件库等）
3. **生成**一条结构化提示词，可直接复制给 Cursor / Claude / Copilot 使用

---

## 什么时候用？

| 场景 | 用这个技能 |
|------|-----------|
| 拿到产品需求，不知道怎么描述给 AI | **用** |
| 想写一个高质量的编码提示词 | **用** |
| 需要把一个复杂需求拆成多条提示词 | **用** |
| PRD 更新了，想生成增量开发的提示词 | **用** |
| 已经有完善的提示词，只想微调 | 不用 |
| 直接让 AI 写代码，不需要提示词工程 | 不用 |

---

## 怎么用？（3 步）

### Step 1：描述你的需求

直接用自然语言告诉我你要做什么，不需要格式化。比如：

> "帮我写个提示词，我要在当前项目新增一个数据导出页面"

### Step 2：回答追问

我会根据你的需求类型，追问你几个关键信息（通常 2-4 个问题）。比如：
- 页面路由是什么？
- 列表数据从哪个接口获取？
- 用什么组件库？

### Step 3：复制提示词

我会生成一条结构化提示词，你直接复制粘贴给 AI 编码工具即可。

## 8 种任务类型 & 真实案例

### 1. PAGE — 页面功能开发

**你的需求：** 新增页面、新增按钮、新增列表、新增表单……

**对话示例：**

> **你：** 帮我写个提示词，我要在当前项目新增一个企微数据导出页面，路由是 /#/weChat_export，上面是筛选区域（企微员工ID多选、起止时间），下面是导出文件列表，导出是异步的，需要每 3 秒轮询一次状态直到完成。接口用 Yapi 的。
>
> **AI 生成：**

```markdown
当前项目需要新增一个独立页面，具体信息如下：

1. 页面路由：/#/weChat_export，页面权限标识：enterprise_weChat_data_export
2. 页面结构分为 2 部分：
   - 上部分：筛选区域
   - 下部分：导出文件展示列表
3. 筛选区域：
   - 企微员工ID列表：支持多选，数据从接口获取
   - 起始时间、结束时间：毫秒时间戳（含）
   - 操作按钮：查询、重置、导出
4. 导出接口中企微员工ID列表不能为空
5. 列表接口支持分页

特殊逻辑：
- 导出操作是异步的，列表需展示：导出中、导出成功、导出失败
- 导出成功时，接口返回的 path 是文件下载地址，操作栏支持下载
- 如果列表中存在「导出中」数据，需要每 3 秒轮询一次
- 所有数据都完成后停止轮询
- 再次点击查询按钮出现「导出中」数据时，重新开启轮询

请根据上述要求开发页面。
```

### 2. UI — UI 还原

**你的需求：** 根据设计稿还原页面、做动画效果、做浮层组件……

**对话示例：**

> **你：** 帮我写个提示词，我需要做一个产品选择浮层，用 vant 组件库。点击「修改可选产品」从底部弹出半屏浮层，里面展示产品列表，最多选 5 个，底部有确认按钮显示已选数量。选完后关闭浮层并更新页面的产品列表。

```markdown
当前需求描述如下：

1. 触发条件：点击「修改可选产品 + 右侧箭头」区域
2. 浮层类型：半屏浮层，从底部弹出
3. 浮层组件库：使用 vant Popup 组件
4. 浮层内容：
   - 产品信息列表，数据通过组件 props 传递的数组展示
   - 每个产品支持勾选
5. 浮层交互：
   - 最多选择 5 个产品
   - 选中状态实时计数
6. 底部按钮：
   - 文案「已选择 N 款产品，确定选择」，N 随选择变动
   - 固定悬浮在浮层底部，iOS 端需支持垫高（safe-area-inset-bottom）
7. 确认后的行为：
   - 关闭浮层
   - 更新页面「产品选择区域」的 products 数据
   - 更新后的产品全部设为选中状态

请根据上述要求进行页面设计还原与功能添加，接口可以先使用 mock 数据。
```

### 3. API — 接口对接

**你的需求：** 对接某个接口、用 MCP 获取 Yapi 文档、多接口联动……

**对话示例：**

> **你：** 帮我写个提示词，我需要对接人脸识别接口，接口是 Yapi 上的 /project/165/api/50766，contextId 前端生成 32 位随机字符串，notifyUrl 和 flowId 不传。返回后把数据缓存，key 是 flowId + contextId。

```markdown
接口文档地址：https://yapi.winbaoxian.cn/project/165/interface/api/50766，使用 MCP 工具获取接口信息。

接口传参说明：
- contextId：前端自动生成 32 位随机字符串，需保证唯一性
- notifyUrl：不传递
- flowId：不传递

接口返回之后的逻辑：
1. 将接口返回的数据保存到缓存中
2. 缓存 key 以 flowId 和 contextId 组合命名
3. 后续在回调页面通过 contextId 读取缓存数据

请根据上述要求对接接口。
```

### 4. ARCH — 技术方案设计

**你的需求：** 设计技术方案、架构重构、解决维护性问题……

**对话示例：**

> **你：** 帮我写个提示词。当前项目的人脸认证函数 gotoFaceCheck 在 20 多个页面被调用，现在要从原生 bridge 改成 H5 第三方认证，如果每个页面都改回调逻辑工作量太大，帮我设计个方案。

```markdown
之前功能描述：
当前文件中的函数 gotoFaceCheck，通过接口获取信息后调用原生 App Bridge 进行身份认证。认证结果通过回调返回，所有业务逻辑都在回调中处理。该函数在项目中有 20+ 处调用。

现在新的需求：
需要通过第三方 H5 页面进行认证。流程变为：跳转 H5 认证页 → 认证完成后带参数跳回 → 根据 URL 参数判断认证结果 → 执行原有回调。

现在的问题是：
如果每个调用页面都单独处理 URL 参数解析和回调逻辑，改动量巨大且不好维护。

请先帮我提供一份合理的技术方案，要求：
1. 考虑复用性：20+ 处调用点尽量不改动
2. 考虑兼容性：原生 Bridge 方式和 H5 方式需要共存
3. 给出具体的改造步骤
```

### 5. REFACTOR — 批量重构

**你的需求：** 域名替换、批量修改配置、多项目同步改造……

**对话示例：**

> **你：** 帮我写个提示词，需要把 18 个前端项目的域名做替换：wyins.net.cn 改成 winbaoxian.cn，wyins.net 改成 winbaoxian.com，同时加 cppoc- 前缀。CDN 和 OSS 不改。每个项目都要新建 poc 分支。

```markdown
需求名称：中邮私有化部署 POC - 前端各项目

需求简介：
需要对当前项目进行改造，核心要求：
1. 新建分支 poc，新增构建命令 npm run build:cppoc
2. 域名替换规则：
   - wyins.net.cn → winbaoxian.cn（加 cppoc- 前缀）
   - wyins.net → winbaoxian.com（加 cppoc- 前缀）
   - 例：manage-planbook-api.winbaoxian.cn → cppoc-manage-planbook-api.winbaoxian.cn
   - 例：claim.wyins.net.cn → cppoc-claim.winbaoxian.cn
3. 页面中「微易」「保险师」字样改成「中邮」，LOGO 图片先注释
4. 例外：CDN 静态资源、OSS 上传不修改

补充规则：
- 建议把项目中的主域名抽取成变量，避免散落硬编码
- 本次改动仅新增测试环境部署，后续部署到中邮环境时还需改一遍

请根据上述要求开发，开发完成后提交到远端 poc 分支，提交信息前缀：中邮私有化部署POC-
```

### 6. DEBUG — Bug 调试

**你的需求：** 排查问题、修复 Bug、定位异常……

**对话示例：**

> **你：** 帮我写个提示词。子应用 preservation_test1 跳转过去能看到页面，但一直反复刷新，停不下来。它的登录体系和主应用不一样，需要二次登录。先不管静默登录，先让流程跑通。

```markdown
当前遇到的问题：

页面路由：/#/new_catalogue_order_manage
子应用名称：preservation_test1

问题现象：子应用页面能看到，但被反复刷新，循环往复，无法正常操作。

背景说明：
- new_catalogue_order_manage 跳转的子应用与当前项目登录体系一致（同域 SSO）
- preservation_test1 对应的子应用登录体系是另一套，需要二次登录

当前排查进展：怀疑是登录态检测导致的循环重定向

请找下问题并修复。先不考虑子应用静默登录，先把流程跑通。
```

### 7. PRD — 需求分析迭代

**你的需求：** 读 PRD 生成开发文档、需求变更后增量开发、对接接口文档……

**对话示例：**

> **你：** 帮我写个提示词。我的 PRD 更新了，之前已经有开发文档和部分代码。我想在现有基础上增量开发，不要全量重写。PRD 在 /docs/prd-v2.md，之前的开发文档在 /docs/dev-doc-v1.md。

```markdown
@pm-md-to-openspec-pipeline @/docs/prd-v2.md

之前的需求开发文档在 /docs/dev-doc-v1.md 中。
需求文档已更新，请按 pm-md-to-openspec-pipeline 执行阶段：

1. 对比之前的需求与现在新的需求文档之间的差异
2. 对新增/改动/优化的功能进行拆分与切片
3. 先告知我你理解的需求变更点，在当前项目中具体有哪些改动与新增
4. 在当前项目已有功能基础上进行增量分析

注意：不要全量重新生成，需要在现有功能基础上识别变更。
```

### 8. MIGRATE — 组件迁移

**你的需求：** 跨项目迁移组件、替换依赖、批量更新版本……

**对话示例：**

> **你：** 帮我写个提示词，我要把 ai-customer-acquisition 项目里的列表组件迁移到当前项目，包括它所有的依赖资源，放在一个独立文件夹下方便维护。

```markdown
在当前项目新增一个独立的页面，页面结构如下：
- 顶部 Tab 切换 + 子菜单 Tab
- 下方列表数据展示

列表展示的功能与另一个项目 ai-customer-acquisition 中的列表组件一致。

要求：
1. 将该组件完整拷贝到当前项目中
2. 组件涉及的所有资源依赖也需搬过去（样式、图片、工具函数等）
3. 将依赖资源与组件放在一个文件夹下（如 src/components/migrated-list/），方便后续维护
4. 点击切换 Tab 或子菜单 Tab 时，列表数据调用接口重新刷新

请完成组件迁移并确保功能正常。
```

## 组合场景：复杂需求怎么拆？

真实工作中，一条需求往往跨多种类型。以下是常见拆分策略：

### 场景：新页面 + UI 还原 + 接口对接

> "我要做一个储蓄专区页面，有 Figma 设计稿，要对接 3 个接口"

**拆分方式：**

| 步骤 | 提示词类型 | 内容 |
|------|-----------|------|
| 1 | `UI` | 根据 Figma 还原页面骨架 |
| 2 | `API` | 对接素材分类接口 + 产品列表接口 |
| 3 | `PAGE` | 组装完整页面逻辑（Tab 切换 + 数据刷新） |

### 场景：PRD 变更 + 增量开发

> "PRD 更新了第二版，之前已经开发了一半"

**拆分方式：**

| 步骤 | 提示词类型 | 内容 |
|------|-----------|------|
| 1 | `PRD` | 对比两版 PRD 差异，生成变更清单 |
| 2 | `PAGE` / `API` | 针对每个变更点生成独立的开发提示词 |

## 常见问题

### Q: 我不知道该提供哪些信息怎么办？

直接说你的需求就行，技能会自动追问缺失的关键信息。通常只需要回答 2-4 个问题。

### Q: 生成的提示词不满意怎么办？

告诉我哪里不对，我会调整。常见调整：
- "太长了，精简一下" → 合并相似步骤
- "缺少 XX 细节" → 补充后重新生成
- "拆成两条" → 按职责拆分

### Q: 能不能一次生成多条提示词？

可以。如果你的需求涉及多个独立功能，告诉我"拆成 N 条提示词"，我会逐条生成。

### Q: 这个技能和直接写代码有什么区别？

这个技能生成的是**给 AI 编码工具看的提示词**，不是代码本身。好处是：
- 提示词可复用（同类型任务换参数就行）
- 团队成员可以用同一套提示词得到一致的代码质量
- 可以提前 review 提示词，避免 AI 编码走偏
