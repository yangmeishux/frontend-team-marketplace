# 前端开发提示词工程 (Frontend Dev Prompt Craft)

> 基于真实前端开发工作沉淀的提示词工程技能  
> 输出契约：见 [`references/output-contract.md`](references/output-contract.md)  
> 提示词模板库：见 [`references/prompt-templates.md`](references/prompt-templates.md)

## When to Activate

- 用户需要将模糊的前端开发需求转化为结构化提示词
- 用户想让 AI 帮忙写一段「给 AI 看的」开发需求描述
- 用户说「帮我写个提示词」「这个需求怎么描述给 AI」
- 用户正在拆解产品需求文档（PRD）为可执行的编码任务
- 用户拿到一个新的前端开发任务，不知道如何开始向 AI 描述

## When NOT to Use

- 用户直接要求写代码（直接编码，不需要提示词工程）
- 非前端开发任务（后端、运维等）
- 用户已经有完善的结构化提示词，只想微调措辞

## Workflow

1. **识别任务类型** — 根据用户描述，判断属于以下 8 种前端任务类型之一（或组合）：

   | 类型代号 | 任务类型 | 典型触发词 |
   |---------|---------|------------|
   | `PAGE` | 页面功能开发 | 新增页面、新增功能、列表页、详情页 |
   | `UI` | UI 还原 | 设计稿、UI稿、Figma、还原页面、布局 |
   | `API` | 接口对接 | 调用接口、MCP、Yapi、接口文档 |
   | `ARCH` | 技术方案设计 | 技术方案、架构设计、怎么实现 |
   | `REFACTOR` | 批量重构 | 域名替换、批量修改、多项目 |
   | `DEBUG` | Bug 调试 | 问题、报错、不执行、反复刷新 |
   | `PRD` | 需求分析迭代 | PRD、需求文档、OpenSpec、增量迭代 |
   | `MIGRATE` | 组件迁移 | 迁移、搬过去、跨项目、依赖 |

2. **收集必要信息** — 根据任务类型，向用户追问缺失的输入（见下方 Input Contract）

3. **加载模板** — Read `references/prompt-templates.md`，找到对应类型的模板

4. **填充生成** — 将用户输入填入模板，生成结构化提示词

5. **自检** — 对照 output-contract 检查生成的提示词是否完整

6. **交付** — 输出结构化提示词 + 使用说明

7. **反馈闭环（可选）** — 交付后追问用户：「这条提示词效果如何？有什么需要改进的？」，然后将反馈追加到 `skill-issues.jsonl` 或 `LEARNINGS.md`

## Input Contract（按任务类型）

### PAGE — 页面功能开发

| 必填 | 说明 |
|------|------|
| 页面位置 | 路由路径 或 文件路径 |
| 功能描述 | 新增什么、怎么交互 |
| 数据来源 | 接口地址 或 mock 说明 |
| 参考信息 | UI稿链接 / 截图 / 类似页面 |

### UI — UI 还原

| 必填 | 说明 |
|------|------|
| 设计稿 | Figma 链接 / 截图 / 布局文字描述 |
| 技术约束 | 使用什么组件库（vant / Element / 自研） |
| 特殊交互 | 动画、滚动、轮播等 |

### API — 接口对接

| 必填 | 说明 |
|------|------|
| 接口文档 | Yapi 链接 / 接口路径 |
| 传参说明 | 前端需要传什么、哪些是自动生成 |
| 返回处理 | 拿到数据后做什么 |

### ARCH — 技术方案设计

| 必填 | 说明 |
|------|------|
| 现状描述 | 当前怎么做的 |
| 新需求 | 需要改成什么 |
| 约束条件 | 兼容性、多页面复用、维护性要求 |

### REFACTOR — 批量重构

| 必填 | 说明 |
|------|------|
| 变更规则 | 旧值 → 新值的映射 |
| 影响范围 | 哪些项目 / 文件类型 |
| 例外说明 | 哪些不需要改（CDN、OSS 等） |

### DEBUG — Bug 调试

| 必填 | 说明 |
|------|------|
| 预期行为 | 应该发生什么 |
| 实际行为 | 实际发生了什么 |
| 复现信息 | URL / 日志 / 截图 |
| 相关代码 | 涉及的文件和方法 |

### PRD — 需求分析迭代

| 必填 | 说明 |
|------|------|
| PRD 文档 | 文档路径或内容 |
| 当前状态 | 已有哪些功能 / 文档 |
| 迭代类型 | 新增需求 / 需求变更 / 增量迭代 |

### MIGRATE — 组件迁移

| 必填 | 说明 |
|------|------|
| 源项目 | 从哪里迁移 |
| 目标项目 | 迁移到哪里 |
| 依赖清单 | 涉及哪些组件和资源 |

## Checkpoints

在以下情况**必须暂停并请求用户确认**，不得自主继续：

- 用户描述的需求涉及多个任务类型组合，需确认优先级
- 生成提示词前，如用户提供的信息不足以填充模板必填项
- 输出可能涉及生产环境操作（部署、批量替换等）

## Output Contract

完整格式见 [`references/output-contract.md`](references/output-contract.md)。

核心要求：
- 每条生成的提示词必须包含：任务类型标签、结构化需求描述、验收标准
- 提示词必须可直接复制粘贴给 AI 编码工具使用
- 如涉及接口，必须包含接口路径和关键传参说明

## Anti-patterns

| 跑偏 | 纠正 |
|------|------|
| 用户只说「帮我写个提示词」就直接生成 | 先追问任务类型和必要信息 |
| 生成模糊的通用提示词 | 必须填充具体的文件路径、接口地址、交互描述 |
| 忽略用户已有的上下文（项目结构、技术栈） | 提示词中明确引用相关文件和路径 |
| 一个提示词塞太多功能 | 单一职责：一条提示词只描述一个可独立完成的任务 |

## 任务类型组合策略

真实工作中，一条需求往往涉及多个类型。常见组合：

| 组合 | 处理方式 |
|------|---------|
| `UI` + `API` | 先 UI 还原，再对接接口；或合并为一条提示词 |
| `PAGE` + `UI` + `API` | 拆分为：UI 骨架 → 接口对接 → 联调三步 |
| `PRD` + `PAGE` | 先需求分析切片，再逐条生成开发提示词 |
| `REFACTOR` + `MIGRATE` | 先迁移，再批量重构 |
| `DEBUG` + `ARCH` | 先定位问题，再设计修复方案 |

## 反馈闭环规则（步骤 7）

每次完成提示词交付后，执行以下闭环：

1. **追问效果**：主动询问用户「这条提示词是否满足需求？有没有需要调整的地方？」

2. **记录问题**：若用户反馈有不足，追加一行到 `skill-issues.jsonl`：
   ```json
   {"date":"YYYY-MM-DD","skill":"frontend-dev-prompt-craft","task_type":"类型代号","symptom":"问题描述","expected":"期望行为","severity":"low|medium|high","source":"user_feedback","converted_to_eval":false,"eval_id":null,"status":"open"}
   ```

3. **沉淀经验**：若有值得复用的洞察，追加到 `LEARNINGS.md`：
   ```markdown
   ## [日期] [任务类型] 简短标题
   **场景**: ...
   **反馈**: ...
   **改进点**: ...
   ```

4. **同步更新**：当 `LEARNINGS.md` 积累 3 条以上同类改进点时，建议用户将这些经验合并回 `SKILL.md` 或 `references/prompt-templates.md`。
